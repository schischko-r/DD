# DD-dev project instructions

## Repository

- Основная рабочая ветка: `dev`.
- Перед началом работы проверяй `git status --short --branch`.
- Не переключай ветки, не делай commit, push или reset --hard без явного запроса пользователя.
- Не трогай `.env`, credential-файлы и секреты.
- Сохраняй существующие незакоммиченные изменения; не перезаписывай их без предупреждения.

## Workflow

1. Сначала изучи структуру проекта, manifest-файлы и существующие реализации.
2. Перед редактированием найди определения символов и их usages.
3. Вноси минимальные изменения только в необходимые файлы.
4. После изменений запускай релевантные тесты, линтер или build.
5. Перед отчётом проверь `git diff --check`, `git diff --stat` и `git status --short --branch`.

## Python

- Используй Python 3.11.
- Для зависимостей и запуска предпочитай существующее виртуальное окружение или `uv`.
- Для Python-изменений запускай соответствующие pytest-тесты.

## Frontend

- Frontend находится в `gravity-app/`.
- После изменений в React-коде запускай релевантные тесты и build из `gravity-app/`.
- Не изменяй generated-файлы и `node_modules` без явной необходимости.

## Git output

- Не коммить и не отправляй изменения на remote без явного запроса.
- В финальном отчёте указывай изменённые файлы, выполненные проверки и их результат.

## Multi-agent workflow

- Kanban board: `dd-dev`.
- Оркестратор сначала уточняет требования, критерии успеха, ограничения и варианты решения.
- `frontend` отвечает за gravity UI и визуальные изменения.
- `backend` отвечает за domain-логику, данные и API.
- `tester` пишет автотесты и запускает проверки.
- `reviewer` независимо принимает результат и возвращает только `APPROVED` или `CHANGES_REQUESTED`.
- Нельзя объявлять задачу `done` до `reviewer: APPROVED`.
- При `CHANGES_REQUESTED` создаются исправления и повторная приёмка.
