# Data-Driven Index

Локальный генератор HTML-отчетов Data-Driven Index из Excel. Сейчас основные сценарии:

- `build_calc_report.py` - собирает один standalone HTML с титульником и деталкой.
- `build_title_from_excel.py` - собирает отдельный standalone HTML только с титульной витриной.

Выходные HTML-файлы открываются как обычные файлы в браузере, без локального сервера: данные встраиваются прямо в `<script type="application/json">`.

## Быстрый запуск

Полный отчет с титульником и деталкой:

```powershell
python build_calc_report.py
```

Явно с параметрами:

```powershell
python build_calc_report.py --input "Расчет_список(1).xlsx" --title-sheet "титул" --detail-sheet "деталка" --output "final_report_from_excel.html"
```

Обычный запуск уже включает AI skill digest в общий пайплайн: генератор скачивает Excel из внутреннего API и использует свежий экспорт в HTML.

```powershell
python build_calc_report.py --output "final_report_from_excel.html"
```

Если нужно не стучаться в API и собрать из локального тестового Excel:

```powershell
python build_calc_report.py --no-download-ai-digest --ai-digest-xlsx "ai_skill_digest_export.xlsx"
```

Только титульная витрина:

```powershell
python build_title_from_excel.py --input "Расчет_список(1).xlsx" --sheet "титул" --output "final_title_from_excel.html"
```

Если нужен файл без `(1)`, передайте его явно через `--input "Расчет_список.xlsx"`.

## `build_calc_report.py`

`build_calc_report.py` - самодостаточный сборщик. Внутри файла зашиты:

- снимок логики `build_title_from_excel.py` для чтения и подготовки данных титульника;
- логика чтения деталки из Excel и сборки вложенной модели продуктов/метрик;
- HTML/CSS/JS-рендер итогового отчета;
- базовый HTML-шаблон отчета.

Файл не импортирует локальные Python-модули и не читает внешний HTML-шаблон во время запуска. Он загружает встроенные строки через `_load_embedded_module(...)`, выполняет их в отдельных namespace и берет оттуда нужные функции. Поэтому для полного отчета на диске нужны только `build_calc_report.py`, входной Excel и Python-зависимости вроде `pandas`/`openpyxl`.

### Вход и выход

По умолчанию:

- вход: `Расчет_список(1).xlsx`;
- лист титульника: `титул`;
- лист деталки: `деталка`;
- период: `II кв. 2026`;
- выход: `final_report_from_excel.html`.

CLI:

```powershell
python build_calc_report.py --input "Расчет_список(1).xlsx" --title-sheet "титул" --detail-sheet "деталка" --period "II кв. 2026" --output "final_report_from_excel.html"
```

Параметры AI skill digest:

```powershell
python build_calc_report.py
python build_calc_report.py --no-update-ai-digest
python build_calc_report.py --ai-digest-token "ваш_токен"
python build_calc_report.py --no-download-ai-digest
python build_calc_report.py --no-download-ai-digest --ai-digest-xlsx "ai_skill_digest_export.xlsx" --ai-product-map "ai_product_mapping.xlsx"
python build_calc_report.py --skip-ai-digest
python build_calc_report.py --refresh-ai-product-map
```

### Шаги сборки

1. `build_combined_data(...)` читает титульный лист через embedded-версию `read_rows(...)`.
2. Титульные строки превращаются в payload через `build_payload(...)`: список строк, уникальные юниты, уникальные типы и средний Data-Driven Index.
3. Детальный лист читается через embedded-версию `build_report_data(...)`.
4. Деталка группируется по `entity_type` и `Продукт`, затем внутри продукта агрегируется по блоку и метрике.
5. Итоговая модель объединяется в один объект:

```python
combined = {**detail_data, "title": title_payload}
```

6. `write_html(...)` передает объединенную модель в embedded-версию `build_embedded_html(...)`.
7. HTML записывается в `final_report_from_excel.html`.
8. В консоль печатается JSON-сводка: путь к HTML, строки деталки, число продуктов, юнитов, агрегированных метрик, строк титульника, юнитов и типов титульника.

### AI skill digest

При обычном запуске сборщик делает `GET /api/skill-digest/export`, скачивает `ai_skill_digest_export.xlsx` из внутреннего API и обновляет светофоры AI-навыков в карточке продукта. Запрос отправляется с заголовком `Authorization: Bearer <token>`; пока токен по умолчанию в коде задан пустой строкой, позже его можно передать через `--ai-digest-token`. Если API недоступен, сборка падает с ошибкой, чтобы не подмешать устаревшую или синтетическую выгрузку. Для локальной проверки можно явно передать `--no-update-ai-digest --ai-digest-xlsx "ai_skill_digest_export.xlsx"`: тогда API не вызывается, а сборщик берет текущий Excel-файл из папки. Старый флаг `--no-download-ai-digest` оставлен как алиас. Чтобы собрать отчет совсем без AI-обогащения, используйте `--skip-ai-digest`.

Экспорт ожидается с колонками:

- `навык`;
- `ключ навыка`;
- `месяц`;
- `продукт`;
- `показатель`;
- `тип` (`светофор` или `текст`);
- `цвет` (`green`, `yellow`, `red`; может быть пустым для текстовых строк);
- `текст`;
- `правило светофора`.

Маппинг ключей навыков:

- `clickstream_funnel` -> `Воронка оформления в СБОЛ`;
- `client_metrics` -> `Навык «Ключевые метрики»`;
- `complaints` -> `Жалобы и обращения`;
- `csi` -> `CSI`;
- `drafts` -> `Черновики`;
- `funnel` -> `Воронка кампейнинга`;
- `pilots` -> `Пилотные кампании`.

Для сопоставления продуктовых названий сборщик создает шаблон `ai_product_mapping.xlsx` с колонками:

- `dd_product`;
- `ai_tool_key`;
- `ai_tool_product name`.

По умолчанию `ai_tool_product name` заполняется тем же названием, что и `dd_product`; если в AI-сервисе продукт называется иначе, нужно поправить только эту колонку. Сборщик берет последнюю дату по паре `продукт + ключ навыка`; цвет строки `тип = светофор` перекрашивает светофор навыка, а строки `тип = текст` попадают в раскрывающийся блок под стрелкой слева от светофора.

### Логика деталки

Деталка собирается встроенной логикой:

- обязательные колонки проверяются до сборки;
- `metric_group`, `metric_name`, футеры и рекомендации чистятся от лишних пробелов;
- `type`/`тип` определяет сущность: `продукт` или `сегмент`;
- значения `value` и `max_value` приводятся к числам;
- строки `BENCHMARKS` делятся на два блока: `Воронка привлечения` и `Воронка оттока`;
- метрики внутри одного продукта и блока суммируются;
- `auto_regularity`, `hyp.ab_tests` и регулярность в блоках привлечения/оттока исключаются из Data-Driven Index;
- TBD-метрики становятся серыми и тоже исключаются из индекса;
- для сегментов скрываются лишние ссылки, AI-навыки и сервисные блоки, которые не должны показываться в сегментной деталке;
- рекомендации берутся из строк расчета, агрегируются по тексту рекомендации, считаются через gap и переводятся в процентные пункты;
- в лендинге рекомендаций сначала показываются top-3 `easy` по убыванию uplift, если `easy` нет - top-3 `medium`.

Индекс в деталке считается на стороне HTML по применимым и не исключенным метрикам:

```text
round(sum(value) / sum(max_value) * 100)
```

## `build_title_from_excel.py`

`build_title_from_excel.py` генерирует отдельную титульную витрину без деталки продукта. Это удобно, когда нужен только список команд с Data-Driven Index, группой, фильтрами и сортировкой.

### Вход и выход

По умолчанию:

- вход: `Расчет_список(1).xlsx`;
- лист: `титул`;
- выход: `final_title_from_excel.html`.

CLI:

```powershell
python build_title_from_excel.py --input "Расчет_список(1).xlsx" --sheet "титул" --output "final_title_from_excel.html"
```

### Обязательные колонки

На листе должны быть:

- `Юнит`;
- `Продукт`;
- `Оценка`;
- `Группа`;
- `type` или `тип`.

Временные Excel-файлы Office с префиксом `~$` отклоняются. Если в обязательных колонках есть пропуски, генератор падает с ошибкой и печатает список проблемных колонок.

### Нормализация данных

Для каждой строки:

- `Юнит`, `Продукт`, `Группа` чистятся от крайних пробелов;
- `type`/`тип` приводится к нижнему регистру и нормализуется в `продукт` или `сегмент`;
- пустой тип считается `продуктом`;
- `Оценка <= 1` считается долей и умножается на 100;
- итоговая оценка округляется, ограничивается диапазоном `0..100` и отображается как процент;
- название `Молодеж` исправляется на `Молодежь`.

Группа около процента берется из Excel без пересчета. Цвета групп:

- `Требуют внимания` - пастельный красный;
- `Развивающиеся` - пастельный оранжевый;
- `Зрелые` - теплый желтый;
- `Лидеры` - пастельный зеленый;
- неизвестная группа - серый fallback.

### Payload

`build_payload(...)` формирует JSON-модель:

- `rows` - строки титульника с `id`, `order`, `unit`, `name`, `score`, `group`, `type`;
- `units` - уникальные юниты по алфавиту;
- `types` - уникальные типы по алфавиту;
- `avgScore` - средний Data-Driven Index по всем строкам.

Эта модель встраивается в HTML в блок:

```html
<script id="data-driven-title-data" type="application/json">
```

### Интерфейс титульника

В итоговом HTML:

- колонка называется `Data-Driven Index`;
- аббревиатуры `DD` в видимой части раскрыты как `Data-Driven`;
- кнопка `Перейти` оставлена, но она серая и неактивная (`disabled`);
- легенда и светофор метрик отсутствуют;
- фильтр `Юнит` содержит `Все юниты` и значения из Excel;
- фильтр `Тип` содержит `Все типы` и значения из Excel;
- фильтры применяются к таблице, счетчикам и среднему индексу;
- если фильтры не дают строк, показывается `Нет данных по выбранным фильтрам`;
- в шапке юнита на узких экранах скрывается счетчик команд, если название юнита не помещается;
- подпись `средний Data-Driven Index` в строке юнита скрывается вместе со счетчиком на узких экранах, остается только процент.

Сортировка:

- `По юнитам`: юниты по алфавиту, внутри каждого юнита продукты/сегменты по алфавиту;
- `По Data-Driven Index`: по убыванию индекса, затем по названию по алфавиту;
- если выбран конкретный юнит, порядок внутри него остается алфавитным по продуктам/сегментам.

После сборки генератор печатает JSON-сводку:

```json
{
  "html": "final_title_from_excel.html",
  "rows": 68,
  "units": 6,
  "types": 2,
  "missing_required_values": 0
}
```
